# Протокол обмена «батарея DJI ↔ зарядный хаб»

Разобрано по двум записям логического анализатора (sigrok/atk_libsigrok 1.0.0, 10 МГц, 2 канала):

- `зарядка_батарей_dji_hub.csv` — батарея в зарядном хабе (35.9 с, идёт зарядка)
- `батарея_включена_по_кнопке_без_дрона.csv` — батарея включена кнопкой, без хаба и дрона

## 1. Физический уровень

- **Интерфейс:** UART, **115200 бод, 8N1**, уровни TTL, idle = high.
  - Битовый интервал в дампе ~8680 нс ⇒ 1/8.68 мкс ≈ 115200.
- **Две сигнальные линии = полудуплекс:**
  - **CH0** — линия запросов: говорит мастер (хаб).
  - **CH1** — линия ответов: говорит батарея.
- Подтверждение: оба канала независимо декодируются в UART, и оба дают валидные DUML-кадры (см. ниже CRC).

## 2. Канальный уровень — DJI DUML

Каждый кадр:

```
off  поле
 0   0x55                  SOF (маркер начала кадра)
 1   len_lo               младший байт длины
 2   (len_hi:2b)|(ver:6b) старшие 2 бита длины + версия протокола (6 бит)
 3   crc8(byte[0..2])      контрольная сумма заголовка (CRC-8, seed 0x77)
 4   SrcID                 отправитель (тип устройства в младших 5 битах)
 5   DstID                 получатель
 6   seq_lo \  16-бит счётчик последовательности (LE)
 7   seq_hi /
 8   attr                  атрибуты: бит 0x40 = запрос, 0x80 = ответ/ACK
 9   CmdSet                набор команд
10   CmdId                 команда
11.. payload               полезная нагрузка
n-2  crc16_lo \ CRC-16 всего кадра кроме этих 2 байт (seed 0x3692, LE)
n-1  crc16_hi /
```

Длина: `len = ((byte[2] & 0x03) << 8) | byte[1]`.
CRC-таблицы — стандартные DJI (см. `duml.py`). **На дампе 1 совпали и CRC-8, и CRC-16 у 2549 кадров** — декод верен, это не совпадение.

## 3. Адресация (наблюдаемая)

| ID    | роль в записи                                     |
|-------|---------------------------------------------------|
| `0x0b`| **Smart Battery** (тип устройства 0x0b)           |
| `0x4b`| мастер/контроллер зарядного хаба (опрашивает)     |
| `0x02`| хост/широковещание (адресат push-статуса батареи) |

- В хабе: `0x4b → 0x0b` (attr `0x40`, запрос) и `0x0b → 0x4b` (attr `0x80`, ответ).
- По кнопке без хаба: батарея сама шлёт `0x0b → 0x02` (push, не по запросу).

## 4. Команды (CmdSet `0x0d` = Smart Battery)

Хаб циклически опрашивает батарею ~14 раз/с. За запись каждая команда повторилась ~505 раз:

| CmdSet:Id | назначение                          | пример payload-ответа |
|-----------|-------------------------------------|------------------------|
| `0d:02`   | **динамический статус** (V/I/T/SoC) | см. §5 |
| `0d:03`   | **напряжения банок** (4S)           | `00 00 04 | fd0e fc0e f80e fe0e` = 3837/3836/3832/3838 мВ |
| `0d:01`   | статика батареи (модель/ёмкости)    | ASCII `WA341`, `PTL` |
| `0d:04`   | серийный номер / штрихкод           | ASCII `8U8PN4RCA0056C` |
| `0d:c0`   | статус-блок                         | `0001 8840 200d 0000 0400 ...` |
| `0d:da`   | статус-блок (V_pack + темпы)        | `0001 9d00 e500 0000 2700 ef3b ...` |
| `0d:11/12/13/26` | служебные (handshake/auth)   | короткие |
| `00:01`   | common GetVersion (CmdSet 0x00)     | серийник + версия |

## 5. Структура payload (офсеты сверены с рабочим парсером `skylab_hub/.../battery.rs`)

Смещения отсчитываются от начала payload; `[0]` = статус (0 = OK).

### `0d:02` — динамический статус

| off | поле                | тип    | значение в дампе |
|-----|---------------------|--------|------------------|
| 0   | статус              | u8     | 0 (OK) |
| 3   | SoC «сырой»         | u8 %   | 59 |
| 6   | **ток**             | i32 LE мА | 0 → +3237 (плюс = заряд) |
| 10  | **полная ёмкость (FCC, изношенная)** | u16 LE мАч | 6198 — делитель для SoC: `SoC[21] ≈ остаток[14]/FCC[10]` = 3464/6198 ≈ 56% |
| 14  | **остаточная ёмк.** | u32 LE мАч | 3348 |
| 18  | **температура**     | u16 LE 0.1°C | 225 = 22.5 °C |
| 20  | число ячеек         | u8     | 4 |
| 21  | **SoC отображаемый**| u8 %   | **55** ← то, что показывает приложение |

### `0d:01` — статика батареи

| off | поле                | тип    | значение в дампе |
|-----|---------------------|--------|------------------|
| 0   | статус              | u8     | 0 |
| 2   | design capacity     | u16 LE мАч | 7100 |
| 6   | **циклы заряда**    | u16 LE | **19** |
| 26..31 | модель (ASCII)   | str    | `WA341` |

### `0d:03` — напряжения ячеек

`[0]`=статус, `[2]`=число ячеек (4), далее N×u16 LE (мВ): `fd0e fc0e f80e fe0e` = 3837/3836/3832/3838 мВ. Напряжение пакета = сумма ячеек ≈ 15348 мВ.

### `0d:c0` — лимиты / статус заряда

| off | поле | значение в дампе |
|-----|------|------------------|
| 0   | статус | 0 |
| 2   | порог напряжения заряда (мВ) | 16520 (≈4×4.13 В, константа) |
| 4   | ёмкость (мАч) | 3360 (≈ остаточная из `0d:02`) |
| 8   | число ячеек | 4 |
| 14  | **флаг «идёт заряд»** | 0 в покое → `0x1800` при заряде |

Поле `[14]` переключается `0 → 0x1800` **ровно в момент появления тока** — подтверждено синхронностью с `0d:02`.

### `0d:da` — термо/зарядная телеметрия

| off | поле | значение в дампе |
|-----|------|------------------|
| 0   | статус | 0 |
| 8   | **напряжение заряда (подаётся с зарядника), мВ** | ≈0 (39) пока заряд не идёт → при заряде ≈ напряжению пакета (±неск. мВ), синхронно с флагом `c0[14]`. ⚠ Бессмысленно как «вольты» на покоящейся батарее (там ~0). |
| 10  | **напряжение пакета, мВ** | 15343 → 15616 (= `0d:02`) — подтверждено суммой банок |
| 2,4,12,14,26,28 | **температуры датчиков (×0.1 °C)** | растут при заряде: напр. `[2]` 15.7→22.0, `[4]` 22.9→23.2, `[12]` 23.7→24.2 |

Реальное поведение по временному ряду: `[8]` сидит на ~39 (≈0) пока заряд не идёт, и **скачком встаёт вровень с `[10]`** (пакетом) в момент включения заряда (флаг `c0[14]` 0→0x1800), ещё до нарастания тока — т.е. это напряжение заряда, подаваемое с зарядника (в CC-фазе почти равно пакету). Температурные поля монотонно растут от нагрева. Точная физическая привязка каждого термодатчика (силовой ключ / ячейки / корпус) не установлена — но `[2]` греется заметно быстрее остальных (вероятно силовой путь заряда).

### `00:01` — GetVersion (версия прошивки/HW)

По разбору заводского DJI-инструмента (`dji_test_userlibs.dll`, Air 3): `00:01` = **GetVersion модуля**, ответ несёт factory-code + версию. В нашем дампе батарея отвечает:
`00 01 | "8U8PN4RCA0056C" (серийник) | 00 00 | 04 00 01 00 24 00 00 2e | 01 00 00 00`

**Версионный дескриптор `04 00 01 00 24 00 00 2e`** (байты `04 · 01 · 24=36 · 2e=46`) присутствует идентично и в `00:01` (off18), и в `0d:01` (off31, сразу после `WA341`+`04`=4S). Это версия/конфигурация прошивки. Точный формат отображения строки (vX.Y.Z) не подтверждён — нужно сверить с тем, что показывает DJI Fly в «О батарее», либо прогнать через dji-firmware-tools.

## 6. Что показали записи

- Батарея от дрона **DJI Mavic 4**: **4S**, модель `WA341`, S/N `8U8PN4RCA0056C`, design capacity 7100 мАч, **циклов заряда 19**, температура 22.5 °C, версионный блок прошивки `04 00 01 00 24 00 00 2e`.
- **Заряд 55%** (поле `0d:02[21]`; сырой SoC 59% в `[3]`). Банки 3.83 В ≈ 55%.
- **Дамп 1:** пойман старт зарядки — ток нарастает 0 → +3.24 А, остаточная ёмкость 3348 → 3373 мАч (фаза CC).
- **Дамп 2:** включённая кнопкой батарея без нагрузки/дрона просто широковещательно объявляет статус (`0x0b → 0x02`), хаб молчит.

## Инструменты

- `duml.py <csv>` — UART-декод обоих каналов + разбор DUML с проверкой CRC + реверс полей `0d:02`.
- `uart_decode.py` — черновой UART-декодер (проверка гипотезы битрейта/полярности).
- `analyze.py` — статистика переходов сигнала (как определялся протокол).

Эталон смещений полей — рабочий парсер `skylab_hub/crates/skylab-hub-core/src/battery.rs`
(`parse_state`/`parse_basic`/`parse_cells`/`parse_serial`), сверен с этим дампом байт-в-байт.

## 7. Live-обмен через USB-UART (`bat_serial.py`) — проверено на железе

Подтверждено на реальной батарее `8U8PN3MCA005U3` (та же модель WA341, циклы 17):

- **Батарею включать НЕ нужно.** Гейдж-MCU запитан от банок и всегда отвечает на UART
  (115200 8N1, idle high). Кнопка не требуется. Достаточно 3 проводов:
  **adapter RX ← линия TX батареи (idle HIGH)**, **adapter TX → линия RX батареи (idle LOW)**, **GND**.
- Мы выступаем мастером-хабом: `0x4b → 0x0b`, attr `0x40`. Запросы — байт-в-байт как у хаба
  (read-payload начинается с `01`). CRC сборки сверены с реальными кадрами (`selftest`).
- Адаптер в тесте: STC USB-Serial (`/dev/ttyACM0`). Любой 3.3 В USB-UART подойдёт.

### Карта ответов набора `0x0d` (перебор 0x00..0xff, read-payload `01`)

Статус-байт ответа `[0]`: `0x00` = OK, **`0xe0` = команда не поддерживается (NACK)**, `0xe3` = иная ошибка.
Из 256 id реально отвечают данными:

| CmdId | назначение | примечание |
|-------|------------|------------|
| `0x01` | статика (модель/design/циклы) | §5 |
| `0x02` | динамический статус V/I/T/SoC | §5 |
| `0x03` | напряжения банок | §5 |
| `0x04` | серийный номер | §5 |
| `0xc0` | лимиты/статус заряда | §5; поле `[8]` зависит от прошивки |
| `0xda` | термо/зарядная телеметрия | §5 |
| `0x17` | `0001 c5e6e568` — 4-байтное значение | nonce/идентификатор? не разобрано |
| `0x20` | 26 байт, в основном нули | статус-блок, не разобрано |
| `0xd2` | **263 байта** (`e14c2a…` + нули) | дамп data-flash/лог? не разобрано |
| `0xe4` | `0001 1417 000000` | пара u16 (0x1714), не разобрано |
| `0x06 0x11 0x19 0x23 0x26 0x54 0xba` | короткие служебные/handshake | не разобрано |

### Диагностический каталог гейджа (cmdset `0x0e`)

Команды `0x0d:0xa2…0xaf` переводят гейдж в режим диагностического стрима (`0x0e:0x21/0x22`):

- **`0x0e:0x21`** — ASCII-список имён всех параметров (магия `d1ed32cf`, токены через `\n`):
  `abm bfm bm cfg cs cfc crm gt1 gt2 gmt gtc1 gtc2 gtf1 gtf2 c1 c2 c3 c4 ccyc hIa cvb dvp dsd drv
  d12v bvel bveh bsox bafsm gpq bsoc bsc bsr cqb cqa cqi bsax bsm cfcc gcur cr ckr balg bcb cp ca
  cpc cac cbo coe …` (`c1..c4`=банки, `ccyc`=циклы, `gt*`=темпы, `gcur`=ток, `cfcc`=полная ёмкость, `bsoc`=SoC).
- **`0x0e:0x22`** — упакованный телеметрический блок (магия `d1ed5f06`, ~98 байт), **НЕ выровнен 1:1 с именами** (смешанные ширины полей). По якорям опознаны: FCC≈6199, остаток 3465, банки `0e0f 0e0f 0a0f 090f`, циклы 17, пакет 15407, темпы 234/240/237. Дублирует штатные `0d:02/03/da` — новых величин не даёт.
- Паспортные с этикетки (**ёмкость 6654 мАч, номинал 14.32 В**) в телеметрии/каталоге **отсутствуют** — рейтинговые константы. Гейдж хранит design 7100 (`0d:01[2]`) и FCC ≈6198-6199 (`0d:02[10]` / `cfcc`).

⚠ Диапазон `0xa0..0xb0` включает debug-стрим — на тестере не нужен, трогать осторожно.
Канарейка (S/N/циклы/SoC до и после полного перебора) — без изменений, перебор read-payload безопасен.

### Инструмент `bat_serial.py`

`sniff [сек]` пассив · `poll [N]` опрос всех read-команд · `read <имя>` одна команда ·
`monitor` живой дашборд · `sweep [set]` перебор id с канарейкой · `send <set> <id> [hex]` произвольная ·
`selftest` сверка CRC. Порт по умолчанию `/dev/ttyACM0`, venv в `.venv/` (pyserial+numpy).
